# ADR-005 — Estratégia de Classificação: Batch vs Individual vs Híbrido

**Status:** Aceito  
**Data:** 2026-06-26  
**Autor:** Agente 3 (Opus)

---

## Contexto

O MailSense precisa classificar e-mails recebidos usando a Claude API. A estratégia de chamada à API determina custo, latência, complexidade e tolerância a falhas do sistema. Volume esperado: ~500 e-mails/dia = ~21/hora = ~3-4 por ciclo de 5 minutos.

---

## Opções Consideradas

### Opção A — Individual (1 e-mail por chamada)

Cada e-mail gera uma chamada independente à API.

**Prós:**
- Falha isolada: um e-mail com problema não afeta os demais
- Retry trivial: task individual é a unidade de falha
- Debugging simples: log 1:1 com cada e-mail

**Contras:**
- Custo alto: 500 chamadas/dia = custo linear de tokens de input (system prompt repetido 500x)
- Rate limiting: risco de atingir limites da Anthropic em picos
- Latência acumulada: cada chamada tem ~1-3s de overhead de rede

**Custo estimado:** ~$0.80-1.20/dia (claude-sonnet-4-6, ~1500 tokens por email × 500 emails × $0.003/1k input tokens)

### Opção B — Batch puro (N e-mails por chamada)

Classifica N e-mails em uma única chamada com prompt estruturado.

**Prós:**
- Custo significativamente menor: system prompt pago uma vez por batch
- Throughput maior: menos overhead de rede
- Custo estimado: ~$0.25-0.40/dia (batch de 5, system prompt amortizado)

**Contras:**
- Falha catastrófica: se a API retorna JSON inválido, todo o batch falha
- Complexidade de prompt: o modelo precisa classificar múltiplos itens mantendo ordem
- JSON parsing mais frágil: arrays grandes têm mais chance de truncamento
- Atribuição ambígua: se o modelo "confunde" e-mails entre si (raro mas possível)

**Custo estimado:** ~$0.15-0.25/dia

### Opção C — Híbrido (Batch primário, Individual como fallback)

Tenta batch de 5. Se o parsing falhar, reclassifica cada e-mail individualmente. Após exaustão individual, marca como FAILED.

**Prós:**
- Custo otimizado no caso feliz (~85% das chamadas)
- Falhas isoladas no fallback individual
- Sem ponto único de falha para o batch completo
- Resiliente a respostas malformadas do LLM

**Contras:**
- Implementação mais complexa (dois caminhos de código)
- Latência aumentada em caso de fallback (~3x mais lenta que batch puro)

**Custo estimado:** ~$0.18-0.30/dia (considerando ~10% de fallback para individual)

---

## Trade-off Analysis

| Critério | Individual | Batch Puro | Híbrido |
|---|---|---|---|
| Custo de API | Alto ($0.80+/dia) | Baixo ($0.15/dia) | Médio ($0.20/dia) |
| Complexidade de implementação | Baixa | Média | Alta |
| Tolerância a falhas | Alta | Baixa | Alta |
| Latência no caminho feliz | Alta | Baixa | Baixa |
| Debugging | Fácil | Difícil | Médio |
| Rate limit safety | Médio | Alto | Alto |

**Volume real** (~500 emails/dia) torna o custo marginal irrelevante para a decisão — a diferença entre as três opções é de centavos por dia. O critério dominante é **robustez** e **facilidade de debugging para portfólio**.

---

## Decisão

**Híbrido com batch de 5 como caminho primário** — exatamente como já desenhado nos stubs existentes.

**Justificativa principal:** O híbrido demonstra engenharia de produção madura (retry inteligente, fallback gracioso) que é mais impressionante para recrutadores do que a simplicidade do individual. O custo marginal entre individual e híbrido é insignificante ($0.10/dia).

**Batch size = 5:** Calibrado para manter o output abaixo de 4096 tokens (margem segura) e garantir que truncamento de resposta seja improvável. Menor que 5 sacrifica muito da economia; maior que 5 aumenta risco de truncamento.

---

## Consequências

1. `classify_pending_batch` usa `SELECT ... FOR UPDATE SKIP LOCKED` para evitar race condition entre workers
2. `ClassificationResult` tem campo `processing_time_ms` e `raw_response` para debugging
3. Fallback individual é acionado quando o JSON do batch não é parseable (não quando há erro de API)
4. Erro de API (429, 500, 503) sempre faz retry com exponential backoff — não aciona fallback
5. Após 2 retries de batch + 2 retries individuais = marcado como FAILED
6. Status `PROCESSING` persiste enquanto a task está ativa — previne double-pick em retries

## Action Items para o Sonnet

- [ ] Implementar `classify_pending_batch` com `select_for_update(skip_locked=True)`
- [ ] Implementar fallback para individual dentro da task batch
- [ ] Usar `exponential_backoff(attempt)` para delays de retry: 30s, 60s, 120s
- [ ] Logar `input_tokens`, `output_tokens`, `processing_time_ms` em cada chamada
- [ ] Criar `_parse_batch_response` com validação de schema por item
