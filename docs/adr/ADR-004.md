# ADR-004: Estratégia de Autenticação — JWT vs Session vs DRF Token

**Status:** Aceito
**Data:** 2026-06-26
**Agente:** 2 — Backend (Opus) via `/architecture`

---

## Contexto

O MailSense precisa autenticar usuários na API REST e também no WebSocket (Django Channels). O sistema tem dois fluxos de auth distintos:

1. **Auth de usuário:** login/registro para acessar a API
2. **OAuth2 Gmail:** conexão com a conta Google para sync de e-mails

A decisão é sobre a estratégia de autenticação da API (fluxo 1). O OAuth2 Gmail (fluxo 2) é separado e obrigatório — não é uma escolha.

## Opções Consideradas

### JWT (djangorestframework-simplejwt)
- Stateless: servidor não mantém estado de sessão
- Access token de curta duração + refresh token
- Pode ser invalidado via blacklist (requer DB) ou expiração natural
- Adiciona complexidade de refresh token logic no cliente
- Padrão moderno para SPAs
- Requer biblioteca extra: `djangorestframework-simplejwt`

### Session-based (Django default)
- Estado mantido no servidor (DB ou cache)
- CSRF obrigatório para requests de formulário
- Não funciona bem com SPA + API desacoplada (CORS + SameSite cookie)
- Nativo do Django — zero dependências extras
- Não adequado para autenticação de WebSocket sem adaptadores

### DRF Token (authtoken)
- Token estático por usuário, armazenado no banco
- Stateful no servidor: `Authorization: Token <hash>`
- Revogação instantânea: delete o token do banco
- Sem expiração nativa — token vive até ser deletado
- Suportado nativamente pelo DRF, sem dependências extras
- Autenticação de WebSocket simples: enviar token no handshake

## Trade-off Analysis

| Critério | JWT | Session | DRF Token |
|---|---|---|---|
| Dependências extras | simplejwt | Nenhuma | Nenhuma (nativo DRF) |
| Revogação imediata | Só com blacklist (DB) | Sim (delete session) | Sim (delete token) |
| Expiração | Configurável | Configurável | Sem expiração nativa |
| WebSocket auth | Middleware customizado | Não adequado | Middleware simples |
| SPA (React) | Ótimo | Problemático (CORS) | Bom |
| Escalabilidade | Stateless | Stateful (DB/cache) | Stateful (DB) |
| Complexidade cliente | Alta (refresh logic) | Baixa | Baixa |
| Volume esperado | < 100 usuários demo | — | Adequado |

## Decisão

**DRF Token Authentication (`rest_framework.authtoken`)**

## Justificativa

1. **Zero dependências:** `authtoken` é parte do DRF, já instalado. Nenhum pacote extra, nenhuma configuração de rotação de tokens, nenhuma refresh logic no cliente React.

2. **Revogação trivial:** `Token.objects.filter(user=user).delete()` invalida instantaneamente. Com JWT sem blacklist, um token comprometido vive até expirar.

3. **WebSocket funciona:** Django Channels autentica via `AuthMiddlewareStack` que suporta DRF tokens nativamente com um middleware de uma linha.

4. **Volume do projeto:** o MailSense é um portfólio demo com < 10 usuários simultâneos esperados. A escalabilidade stateless do JWT não tem valor prático aqui.

5. **Complexidade de JWT desnecessária:** refresh tokens exigem lógica no cliente (interceptors, retry logic, 401 handling). O `useAuthStore` do Zustand fica mais simples com um token estático.

6. **Segurança adequada:** HTTPS em produção + token no header `Authorization` é suficiente. Sem expiração é aceitável para portfólio; se necessário, uma task Celery pode rotacionar tokens periodicamente.

## Consequências

- `INSTALLED_APPS` inclui `rest_framework.authtoken`
- `DEFAULT_AUTHENTICATION_CLASSES` = `[TokenAuthentication]`
- Login retorna `{"token": "...", "user_id": N}`
- Cliente React guarda token no `localStorage` via Zustand persist
- Axios interceptor injeta `Authorization: Token <token>` em todos os requests
- WebSocket handshake: token no query param ou header — implementar no `AuthMiddlewareStack`
- Logout = `DELETE /api/v1/auth/logout/` que deleta o token do banco

## Sobre o OAuth2 Gmail (separado desta decisão)

O OAuth2 com Google é para sync de e-mails, não para login no MailSense. O fluxo é:
1. Usuário já autenticado (DRF Token) chama `POST /api/v1/auth/gmail/connect/`
2. Recebe `auth_url` → redireciona para Google
3. Google retorna para `/api/v1/auth/gmail/callback/?code=...`
4. Tokens OAuth armazenados em `UserProfile.gmail_access_token/refresh_token`
5. Os tokens Gmail são usados apenas pelo `GmailService` no backend — nunca expostos ao frontend

## Alternativas Rejeitadas

- **JWT:** complexidade de refresh logic desnecessária para o volume e contexto do projeto
- **Session:** incompatível com SPA desacoplada em domínio diferente (CORS, SameSite=Strict)
