# ADR-003: Framework de API Backend — DRF vs Django Ninja vs FastAPI

**Status:** Aceito
**Data:** 2026-06-26
**Agente:** 2 — Backend (Opus) via `/architecture`

---

## Contexto

O MailSense precisa de uma camada de API REST que sirva o frontend React. O Django 5 foi escolhido como framework web base (ORM, admin, migrations, auth, Celery integration). A decisão é qual biblioteca construir a API em cima do Django.

Os critérios relevantes para este projeto:
- Integração com Django ORM, migrations, admin
- Velocidade de desenvolvimento (projeto de portfólio, solo dev)
- Ecossistema de filtros, paginação, serialização
- Compatibilidade com Django Channels (WebSocket no mesmo projeto)
- Geração automática de schema OpenAPI para docs
- Maturidade e recognição por recrutadores

## Opções Consideradas

### Django REST Framework (DRF)
- Biblioteca mais estabelecida para APIs Django (~4M downloads/mês)
- ModelSerializer gera serializers a partir dos models automaticamente
- django-filter integração nativa
- Throttling, pagination, permissions embutidos
- drf-spectacular para OpenAPI
- Verboso em alguns casos (serializers longos)
- Nenhuma tipagem nativa — sem type hints nos serializers

### Django Ninja
- API moderna para Django com type hints nativos (Pydantic v2)
- Geração automática de OpenAPI via tipos
- Muito mais conciso que DRF
- Ainda imaturo (~150K downloads/mês)
- Menos recursos out-of-the-box (sem filterset nativo, pagination manual)
- Menos conhecido por recrutadores

### FastAPI (standalone, sem Django ORM)
- O mais performático dos três
- Type hints, Pydantic v2, async nativo
- Exigiria trocar para SQLAlchemy ou Tortoise ORM
- Perde: Django admin, Django migrations, Django auth, django-celery-beat
- Overhead de manter dois frameworks em paralelo (Django para Celery + FastAPI para API)
- Não faz sentido arquitetural dado que já escolhemos Django

## Trade-off Analysis

| Critério | DRF | Django Ninja | FastAPI |
|---|---|---|---|
| Integração Django | Nativa | Nativa | Indireta |
| Velocidade de dev | Alta (ModelSerializer) | Alta (schemas Pydantic) | Média (ORM diferente) |
| Filtros e paginação | django-filter pronto | Manual | Manual |
| Type safety | Baixa | Alta (Pydantic) | Alta (Pydantic) |
| Maturidade | 12+ anos | 3 anos | 6 anos |
| Recognição portfólio | Muito alta | Média | Muito alta |
| OpenAPI automático | drf-spectacular | Nativo | Nativo |
| Async support | Parcial | Parcial | Total |

## Decisão

**Django REST Framework (DRF)**

## Justificativa

1. **Alinhamento com Django:** DRF foi projetado para Django. Compartilha migrations, admin, auth e signals sem adaptadores.

2. **Velocidade com ModelSerializer:** Os models já estão definidos. DRF gera 80% do código dos serializers automaticamente, acelerando a implementação do Agente 2 (Sonnet).

3. **django-filter nativo:** Os requisitos incluem filtros compostos (`?category=support,billing&priority=critical`). O `django-filter` integra com DRF via um atributo de classe — zero configuração extra.

4. **Portfólio:** DRF é o padrão de mercado para APIs Django. Recrutadores reconhecem imediatamente; Django Ninja ainda não tem essa penetração.

5. **Custo do Ninja:** Os ganhos de type safety do Django Ninja não compensam a imaturidade do ecossistema (sem filterset nativo, pagination manual, menos plugins). Para um projeto solo de portfólio, a verbosidade extra do DRF é gerenciável.

6. **FastAPI eliminado:** Exigiria abandonar Django ORM + admin + migrations, que são centrais para Celery Beat, seed management commands e admin de debug.

## Consequências

- Serializers divididos em `list` (leve) e `detail` (completo) para controlar payload
- `drf-spectacular` gera OpenAPI schema → Swagger em `/api/docs/` e Redoc em `/api/redoc/`
- Throttling, paginação e permissões configurados em `settings.REST_FRAMEWORK`
- Se o projeto evoluir para alta concorrência async, reavaliar com ADR-revisão

## Alternativas Rejeitadas

- **Django Ninja:** ecossistema imaturo, filtros manuais, menor recognição para portfólio
- **FastAPI:** perda total da integração Django; custo arquitetural injustificado dado a escolha prévia de Django+Celery
