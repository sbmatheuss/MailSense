MailSense — Prompt de Arquitetura Completa
Prompt master para construção do projeto MailSense: plataforma de resumo e classificação inteligente de e-mails/tickets com IA.


________________


Contexto do Projeto
Você é o arquiteto principal do MailSense, uma plataforma web fullstack que conecta com Gmail (ou simula uma inbox), classifica e-mails automaticamente por categoria, prioridade e sentimento usando LLM, sugere respostas, identifica urgências e apresenta tudo em um dashboard inteligente.


O projeto será desenvolvido com a seguinte stack:


* Backend: Python 3.12, Django 5.x, Django REST Framework, Celery, Redis, PostgreSQL, Django Channels (WebSocket)
* Frontend: React 18 + TypeScript, Vite, TailwindCSS, Shadcn/UI, Recharts, React Query (TanStack Query)
* IA: Anthropic Claude API (claude-sonnet-4-6)
* Infra: Docker, Docker Compose, Nginx, GitHub Actions (CI/CD)
* Deploy: Railway ou Fly.io


O público-alvo é recrutadores e tech leads avaliando portfólio. O projeto precisa ser funcional, bem documentado, com modo demo (sem precisar conectar Gmail), e visualmente impressionante.

Estratégia de Modelos — Opus vs Sonnet
O projeto utiliza dois modelos Claude de forma estratégica. A regra geral é: Opus pensa, Sonnet executa.
Quando usar Opus (claude-opus-4-6)
Opus é o modelo mais capaz em raciocínio profundo, decisões arquiteturais e resolução de problemas complexos. Use nos momentos em que uma decisão errada custaria horas de retrabalho:


Tarefa
	Por que Opus
	Definir arquitetura inicial do projeto
	Decisões estruturais afetam tudo depois — models, relações, indexes, separação de apps
	Desenhar o schema do banco de dados
	Relações entre Email, Classification, ActionLog, threads — precisa pensar em queries futuras
	Criar os prompts de classificação da IA
	O prompt é o coração do produto — tom, formato JSON, edge cases, instruções de prioridade
	Projetar o sistema de filas e tasks Celery
	Orquestração batch vs individual, retry, dead letter, schedule — lógica não-trivial
	Arquitetar o fluxo OAuth2 com Gmail
	Segurança, refresh de tokens, escopos, tratamento de revogação
	Resolver bugs complexos ou de integração
	Quando o erro cruza camadas (frontend ↔ API ↔ Celery ↔ LLM)
	Code review final antes de deploy
	Visão holística de segurança, performance, edge cases
	Escrever o README e documentação técnica
	Narrativa, decisões técnicas, comunicação clara para recrutadores
	Definir o design system e UX principal
	Paleta, tipografia, hierarquia visual, experiência do dashboard
	Quando usar Sonnet (claude-sonnet-4-6)
Sonnet é rápido, eficiente e excelente para execução. Use para tudo que já tem direção clara definida pelo Opus:


Tarefa
	Por que Sonnet
	Implementar models Django a partir do schema
	O schema já foi decidido — é execução
	Escrever serializers e views DRF
	Padrão repetitivo com variações — Sonnet faz bem e rápido
	Implementar componentes React individuais
	EmailList, Badge, Chart — componentes isolados com spec clara
	Escrever CSS/Tailwind e estilização
	Aplicar o design system já definido
	Criar os endpoints REST
	CRUD, filtros, paginação — padrão Django REST
	Implementar tasks Celery individuais
	A arquitetura já existe, cada task é implementação direta
	Escrever testes unitários e de integração
	Testes seguem os models e endpoints já criados
	Gerar dados de seed/demo
	Criar fixtures e e-mails fictícios — volume com padrão
	Configurar Docker, CI/CD, Makefile
	Receitas conhecidas, pouca decisão nova
	Corrigir bugs simples e ajustes pontuais
	Erros isolados em um arquivo ou componente
	Refatorações locais
	Extrair função, renomear, reorganizar imports

Regras de troca entre modelos
1. Comece cada fase no Opus — ele pensa, planeja e decide
2. Mude para Sonnet quando o plano estiver claro — ele executa rápido e barato
3. Volte para Opus se encontrar um problema inesperado — bug cross-layer, decisão arquitetural não prevista, refatoração grande
4. Use Opus para toda revisão final — antes de commitar uma fase, Opus faz code review e identifica falhas que Sonnet pode ter deixado passar
5. Nunca use Opus para tarefas repetitivas — se você está escrevendo o quinto serializer, é trabalho de Sonnet
6. Nunca use Sonnet para decisões que afetam múltiplos arquivos — se a mudança cascateia, Opus precisa pensar o impacto antes

Sub-Agentes de Desenvolvimento
O desenvolvimento será dividido entre sub-agentes especializados. Cada agente tem responsabilidades claras e deve seguir as diretrizes abaixo.


________________


Agente 1 — Arquiteto de Infraestrutura
Responsabilidade: Setup inicial do projeto, containerização, CI/CD e configuração de ambientes.


Tarefas:


1. Criar docker-compose.yml com serviços: web (Django), worker (Celery), beat (Celery Beat), redis, postgres, frontend (React/Vite)
2. Criar Dockerfile para backend (Python 3.12-slim) e frontend (Node 20-alpine)
3. Criar .env.example com todas as variáveis necessárias:
   * DATABASE_URL, REDIS_URL, DJANGO_SECRET_KEY, DJANGO_DEBUG
   * ANTHROPIC_API_KEY
   * GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET
   * ALLOWED_HOSTS, CORS_ALLOWED_ORIGINS
4. Configurar config/settings/base.py, dev.py, prod.py com split de ambientes
5. Configurar config/celery.py com autodiscovery de tasks
6. Criar nginx.conf para servir frontend + proxy para API
7. Criar .github/workflows/ci.yml:
   * Lint (ruff + eslint)
   * Testes (pytest + vitest)
   * Build das imagens Docker
   * Deploy automático para staging
8. Criar Makefile com comandos úteis:
   * make up, make down, make migrate, make seed, make test, make lint


Diretrizes:


* Usar multi-stage build nos Dockerfiles
* Healthchecks em todos os serviços
* Volumes nomeados para persistência do banco
* Hot reload funcionando em dev para backend e frontend

SYSTEM PROMPT:


Você é um assistente de triagem de e-mails corporativo. Sua tarefa é analisar e-mails


e retornar uma classificação estruturada em JSON.


Regras:


- Analise o subject e body completos


- Considere o tom, urgência implícita e palavras-chave


- Para prioridade CRITICAL: risco financeiro, legal, segurança, SLA estourado, cliente VIP irritado


- Para prioridade HIGH: reclamação direta, prazo próximo, bug em produção


- Para prioridade MEDIUM: solicitação normal, dúvida técnica, follow-up


- Para prioridade LOW: newsletter, info geral, agradecimento, notificação automática


- O summary deve ter no máximo 2 frases, capturando a essência


- A suggested_reply deve ser profissional, empática e resolver o problema


- key_topics: extraia de 2 a 5 tópicos principais como tags

Instruções para os Sub-Agentes
Cada sub-agente deve seguir estas regras:


1. Sempre criar arquivos completos e funcionais — nada de "# TODO implement"
2. Seguir o padrão definido neste documento — nomes de variáveis, estrutura de pastas, convenções
3. Incluir docstrings em todas as classes e métodos públicos
4. Type hints em todo código Python
5. TypeScript strict no frontend — sem any
6. Commits semânticos: feat:, fix:, docs:, test:, chore:
7. Tratamento de erros em todas as chamadas externas (Gmail API, Claude API, banco)
8. Logging estruturado com structlog ou logging configurado
9. Sem secrets hardcoded — tudo via variáveis de ambiente
10. Código limpo — sem imports não usados, sem código comentado, sem prints soltos
11. Invocar skills antes de decidir — toda decisão arquitetural passa por /architecture, todo design por /frontend-design, todo plano de testes por /testing-strategy. Não "pular" a skill porque "já sabe a resposta"
12. Documentar com skills — usar /documentation para README, runbooks e architecture docs. Docstrings inline em toda classe e método público
13. Code review obrigatório — ao final de cada fase, rodar /code-review no código produzido antes de seguir para a próxima fase
14. Deploy checklist obrigatório — na Fase 6, rodar /deploy-checklist antes de qualquer deploy
15. Registrar débitos — se precisar cortar corners por tempo, registrar como tech debt para revisão via /tech-debt na Fase 5

Fluxo de troca de modelo durante o desenvolvimento:
1. Abra sessão com Opus → planeje a fase atual


2. Copie as decisões/plano do Opus


3. Troque para Sonnet → implemente passando o plano como contexto


4. Se encontrar bug complexo → volte para Opus com o erro


5. Opus resolve → Sonnet aplica o fix


6. Fim da fase → Opus faz code review final
