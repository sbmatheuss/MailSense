# ADR-006 — Provider de LLM: Claude API vs OpenAI vs Multi-provider

**Status:** Aceito  
**Data:** 2026-06-26  
**Autor:** Agente 3 (Opus)

---

## Contexto

O núcleo do MailSense é a classificação de e-mails com LLM. A escolha do provider define custo, qualidade de output, complexidade de integração e narrativa do portfólio. O projeto já tem `ANTHROPIC_API_KEY` como variável de ambiente definida.

---

## Opções Consideradas

### Opção A — Claude API (Anthropic)

**Modelo:** `claude-sonnet-4-6` para classificação, `claude-opus-4-6` para decisões arquiteturais.

**Prós:**
- Excelente aderência a JSON — crítico para parsing sem erros em batch
- Contexto longo (200k tokens) — lida bem com e-mails volumosos
- Raciocínio superior em nuances de tom e sentimento vs. GPT-3.5/4o-mini
- Coerência narrativa: o projeto é avaliado por quem usa Claude — demonstra domínio do ecossistema
- SDK Python maduro (`anthropic>=0.40.0` já no requirements)
- Preço competitivo: claude-sonnet-4-6 tem ótimo custo/qualidade

**Contras:**
- Menos onipresente que OpenAI no mercado
- Função calling menos documentada que GPT function calling (mas JSON nativo é equivalente)

### Opção B — OpenAI (GPT-4o / GPT-4o-mini)

**Prós:**
- Maior familiaridade do mercado
- Function calling nativo bem estabelecido
- GPT-4o-mini muito barato

**Contras:**
- GPT-4o-mini tem qualidade inferior para nuances de sentimento/prioridade
- GPT-4o custa mais que claude-sonnet-4-6 para qualidade equivalente
- Não alinha com a narrativa do portfólio (MailSense usa Claude = demonstra Anthropic stack)
- JSON adherence ocasionalmente inferior em prompts complexos de batch

### Opção C — Multi-provider (Claude + OpenAI fallback)

**Prós:**
- Alta disponibilidade: se Anthropic cair, usa OpenAI
- Comparação de outputs possível

**Contras:**
- Complexidade dobrada: dois SDKs, dois modelos de custo, dois formatos de resposta
- Os outputs divergem — comparações são inconsistentes para métricas
- Desnecessário para volume de portfólio (<1000 emails/dia)
- Não agrega valor para recrutadores avaliando o projeto

---

## Decisão

**Claude claude-sonnet-4-6** como único provider para classificação.

**Justificativa:**
1. O MailSense é um portfólio *sobre* IA com Claude — usar outro provider seria incoerente com a narrativa
2. A qualidade de JSON adherence do claude-sonnet-4-6 reduz erros de parsing, que é o principal risco operacional do batch
3. Para o volume esperado (<500 emails/dia), custo e disponibilidade são critérios secundários
4. Multi-provider geraria complexidade que não seria avaliada positivamente — parece over-engineering para portfólio

---

## Configuração do Modelo

```python
# Classificação em produção
CLASSIFICATION_MODEL = "claude-sonnet-4-6"

# Parâmetros de chamada
MAX_TOKENS_SINGLE = 1024    # Suficiente para 1 classificação JSON
MAX_TOKENS_BATCH = 4096     # Suficiente para 5 classificações JSON
TEMPERATURE = 0             # Zero: saída determinística para JSON
BODY_TRUNCATION_CHARS = 2000  # Controle de custo: corpo truncado a 2000 chars
```

---

## Consequências

1. `LLMService` usa apenas `anthropic.Anthropic()` — sem abstração multi-provider
2. `temperature=0` em todas as chamadas de classificação — outputs consistentes
3. Body truncado a 2000 chars **antes** de montar o prompt — economia real de ~40% nos tokens de input
4. `max_tokens` calculado como `1024 * len(batch)` não excedendo 4096 — evita truncamento
5. Logging de `usage.input_tokens` e `usage.output_tokens` de cada resposta — métricas de custo visíveis no dashboard futuro

---

## Action Items para o Sonnet

- [ ] `LLMService` sem abstração provider — acesso direto ao `anthropic.Anthropic()`
- [ ] Truncar body em `body[:2000]` antes de `_build_user_prompt`
- [ ] Usar `temperature=0` em todas as chamadas `client.messages.create()`
- [ ] Logar `message.usage.input_tokens` e `message.usage.output_tokens`
- [ ] Calcular `max_tokens = min(1024 * len(batch), 4096)` dinamicamente
