# ADR-002: Estratégia Docker — Multi-stage Build

**Status:** Aceito
**Data:** 2026-06-26
**Agente:** 1 — Infraestrutura (Opus)

---

## Contexto

Os Dockerfiles precisam atender dois contextos distintos:
- **Dev:** hot reload, debugging, montagem de volumes, dependências de desenvolvimento
- **Prod:** imagem enxuta, segura, sem ferramentas de compilação expostas

A escolha afeta tamanho das imagens, velocidade de build, segurança e complexidade de manutenção.

## Opções Consideradas

| Critério | Multi-stage | Single-stage dev | Dois Dockerfiles separados |
|---|---|---|---|
| Tamanho imagem prod | ~150 MB | ~800 MB+ | ~150 MB |
| Segurança | Sem compiladores na imagem final | Build tools expostos | Depende |
| Manutenção | Um arquivo por serviço | Simples, mas inseguro | Dois arquivos, duplicação |
| Cache de layers | Melhor (deps → código) | Igual | Igual |
| `docker-compose` dev | `target: development` | Direto | `dockerfile: Dockerfile.dev` |
| Portabilidade CI | Alta | Alta | Média (duas tags) |

## Decisão

**Multi-stage build** em dois Dockerfiles (`Dockerfile.backend`, `Dockerfile.frontend`), com estágios nomeados reutilizados pelo `docker-compose`:

### Dockerfile.backend
```
FROM python:3.12-slim AS base       # deps comuns + sistema
FROM base AS development            # + dev tools, watchfiles
FROM base AS production             # + gunicorn, sem gcc/pip
```

### Dockerfile.frontend
```
FROM node:20-alpine AS deps         # npm ci
FROM deps AS builder                # vite build
FROM nginx:alpine AS production     # apenas /dist + nginx.conf
```

### Orquestração

| Arquivo | Uso | Target |
|---|---|---|
| `docker-compose.yml` | Dev local | `development` (backend), `deps` (frontend) |
| `docker-compose.prod.yml` | Produção | `production` (ambos) |

## Serviços e Networks

```
Serviços:        Dependências:
web            → postgres, redis
worker         → postgres, redis
beat           → postgres, redis
nginx          → web (prod only)
frontend       → (standalone dev server)
postgres       → (volume: postgres_data)
redis          → (volume: redis_data)
```

Todos os serviços na mesma network default do Compose (`mailsense_default`). Sem network customizada necessária para v1.

## Volumes

| Volume | Propósito |
|---|---|
| `postgres_data` | Persistência do banco entre restarts |
| `redis_data` | Persistência de filas entre restarts |
| `frontend_node_modules` | Cache de node_modules no container |
| `./backend:/app` (dev only) | Hot reload do código Python |
| `./frontend:/app` (dev only) | Hot reload do código React |

## Healthchecks

Todos os serviços com healthcheck:
- `postgres`: `pg_isready`
- `redis`: `redis-cli ping`
- `web` (prod): `GET /api/health/`
- `nginx` (prod): `wget -qO- /health`

## Justificativa

O delta de complexidade do multi-stage (um `FROM` extra por Dockerfile) é pequeno. Os ganhos são:
1. **Segurança:** gcc, pip, compiladores não existem na imagem de produção
2. **Tamanho:** ~5× menor que single-stage com build tools
3. **Cache:** layer de dependências fica separada do código — rebuilds são rápidos
4. **Portfólio:** recrutadores que lêem o Dockerfile reconhecem a boa prática

## Consequências

- CI precisa especificar `--target production` no build step
- `docker-compose.yml` usa `target: development` para backend
- Frontend em dev usa `node:20-alpine` diretamente (sem Nginx) para hot reload via Vite
