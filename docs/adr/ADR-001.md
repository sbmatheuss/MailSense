# ADR-001: Estrutura do Repositório — Monorepo vs Poly-repo

**Status:** Aceito
**Data:** 2026-06-26
**Agente:** 1 — Infraestrutura (Opus)

---

## Contexto

O MailSense possui dois artefatos distintos — backend Django e frontend React — que precisam ser desenvolvidos, testados e deployados de forma coordenada. A escolha da estrutura afeta CI/CD, onboarding, compartilhamento de tipos e complexidade operacional.

O projeto é desenvolvido por um único desenvolvedor e tem como objetivo principal ser portfólio para recrutadores.

## Opções Consideradas

| Critério | Monorepo | Poly-repo |
|---|---|---|
| Onboarding | Um único `git clone` + `make up` | Dois repos, dois setups |
| CI/CD | Uma pipeline com path filters | Pipelines independentes |
| Compartilhamento de tipos | Fácil (pasta `/shared` futura) | Requer pacote publicado |
| Isolamento de deploy | Path filters no workflow | Natural por repo |
| Portfólio (recrutadores) | Um link, tudo visível | Dois links, fragmentado |
| Conflitos de branch | Possíveis entre frontend/backend | Isolados |
| Versioning independente | Requer tags customizadas | Natural via releases |

## Decisão

**Monorepo** com estrutura:
```
mailsense/
├── backend/     # Django + Celery
├── frontend/    # React + Vite
├── nginx/       # Configuração do reverse proxy
├── docs/        # ADRs e documentação de arquitetura
└── .github/     # CI/CD com path filters
```

## Justificativa

Para um projeto de portfólio com desenvolvedor solo:

1. **Experiência do recrutador:** um único link GitHub mostra a stack completa sem navegação entre repositórios
2. **Setup instantâneo:** `git clone && make up` → ambiente completo em execução
3. **CI/CD simplificado:** uma pipeline com `paths:` filters evita runs desnecessários sem duplicar workflows
4. **Sem overhead de sincronização:** backend e frontend evoluem na mesma branch, sem versioning cross-repo

O custo (possíveis conflitos de merge entre áreas) é insignificante com um desenvolvedor.

## Consequências

- `.github/workflows/ci.yml` usa `paths:` filters para jobs seletivos por área
- `Makefile` unificado na raiz com targets para ambos (`make test-back`, `make test-front`)
- Dockerfiles separados por serviço (`Dockerfile.backend`, `Dockerfile.frontend`)
- Se o projeto crescer para múltiplos desenvolvedores, reavaliar com ADR-revisão

## Alternativas Rejeitadas

- **Poly-repo:** overhead de sincronização e fragmentação da experiência de portfólio superam os benefícios de isolamento para este contexto
